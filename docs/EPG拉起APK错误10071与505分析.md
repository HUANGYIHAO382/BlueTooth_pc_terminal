# EPG 拉起 APK 错误分析（10071 / 505 / HTTP 500）

> 分析对象：`docs/openApp.jsp`、抓包文件 `docs/医鸿错误代码10071抓包.pcapng`  
> 目标包名：`com.example.myapplication`  
> 目标入口：`com.example.myapplication.MainActivity`  
> 分析日期：2026-07-07

---

## 1. 结论（先看这里）

| 现象 | 抓包真实原因 | 是否已执行到 APK 拉起 |
|------|-------------|----------------------|
| 机顶盒显示 **10071**（页面访问超时/无响应） | EPG 服务器对 `openApp.jsp` 返回 **HTTP 500** | **否** |
| 你提到的 **505** | 抓包中 **没有 HTTP 505**；最接近的是 **500 Internal Server Error** | **否** |

**一句话：问题出在 EPG 服务端 JSP 页面执行失败，JavaScript 里的 `STBAppManager.startAppByIntent()` 根本没有机会运行。**

在 JSP 500 修好之前，改包名、改类名、改 `appId` 都不会生效。

## 3. 为什么 openApp.jsp 会 500？

当前 `openApp.jsp` 强依赖华为 IPTV EPG 平台的目录结构与 Java 类库，**不能直接放到任意路径就运行**。

### 3.1 部署路径与 include 路径不匹配（高概率）

抓包显示实际访问路径：

```text
/EPG/jsp/CNEPG/en/openApp.jsp
```

而 JSP 内引用的相对路径是按另一套目录写的：

```jsp
<%@ include file="../../../hwdatajsp/checkReport.jsp" %>
<script src="../../../js/report.js"></script>
<script src="../../../js/WkEpg.js"></script>
<script src="../../../config/config.js"></script>
<jsp:include page="../../../util/passwordEncryption.jsp" ... />
```

若从 `/EPG/jsp/CNEPG/en/` 推算，`../../../` 会指向 `/EPG/`，需要该位置存在：

- `hwdatajsp/checkReport.jsp`
- `js/report.js`、`js/WkEpg.js`
- `config/config.js`
- `util/passwordEncryption.jsp`

**任一文件缺失或路径错误，JSP 编译/执行就会抛异常 → HTTP 500。**

### 3.2 依赖华为 EPG Java 类（高概率）

```jsp
<%@ page import="com.huawei.iptvmw.epg.bean.info.UserProfile"%>
<%@ page import="com.huawei.iptvmw.epg.bean.MetaData"%>
UserProfile userProfile = new UserProfile(request);
```

这些类只在 **华为 IPTV EPG 应用服务器（Tomcat/WebLogic 等）** 的 `WEB-INF/lib` 中存在。  
若 JSP 被放到错误容器、或类路径未加载，会在 **服务端** 直接 500。

### 3.3 passwordEncryption.jsp 加密用户 ID（中概率）

```jsp
<jsp:include page="../../../util/passwordEncryption.jsp">
    <jsp:param name="userID" value="<%=userId%>" />
    ...
</jsp:include>
```

页面依赖 `encryurID` 变量。若 `passwordEncryption.jsp` 不存在或加密逻辑异常，会导致 JSP 执行失败。

### 3.4 URL 参数重复（低～中概率）

抓包 URL：`userId=075540312817&&userId=075540312817`（双 `&`、双 `userId`）。

部分 EPG 解析逻辑可能对异常 query 不健壮，建议 EPG 配置侧修正链接生成方式。

## 4. openApp.jsp 代码层面的第二个问题（JSP 修好后才会暴露）

即使 JSP 500 修好，当前脚本还有 **包名判断逻辑未同步修改** 的问题。

### 4.1 `judgeApk()` 仍检查旧包名

```javascript
// 第 46 行：判断的是酷喵旧包，不是你改的新包
function judgeApk(){
    var value = STBAppManager.isAppInstalled("com.video.ytlook_SZDX");
    return value;
}
```

而你实际拉起的是：

```javascript
appPkg=com.example.myapplication
appName: "com.example.myapplication"
className: "com.example.myapplication.MainActivity"
```

**后果：**

| 机顶盒是否安装旧酷喵 APK | 实际走哪条分支 | 问题 |
|--------------------------|----------------|------|
| 未安装 `com.video.ytlook_SZDX` | 走应用商城下载分支 | 需要 `appId=479` 在商城后台已绑定新包名 |
| 已安装旧酷喵 | 走「已预装」直接 `startAppByIntent` | 会尝试启动 `com.example.myapplication`，与判断对象不一致 |

### 4.2 与本仓库 APK 包名不一致

本仓库 `app/build.gradle` 实际配置为：

```gradle
applicationId "com.iknet.bloodmeasuredemo"
```

`AndroidManifest.xml` 中 `MainActivity` 全类名为：

```text
com.iknet.bloodmeasuredemo.MainActivity
```

若你安装的是本仓库编译的 APK，但 JSP 写的是 `com.example.myapplication`，**即使 JSP 正常也会启动失败**。

### 4.3 应用商城 appId 可能未更新

未预装分支使用：

```javascript
extraKey: "jumpId=8&appPkg=com.example.myapplication&appId=479&account=null"
```

`appId=479` 很可能仍是旧应用（酷喵）在商城后台的登记 ID。  
新包名必须在 **应用商城运营后台** 重新上架/绑定，否则商城无法下载正确 APK。

## 5. 解决方案（按优先级）

### 阶段 A：先修 HTTP 500（必须，否则一切无效）

#### A-1. 确认 JSP 部署位置与原厂模板一致

向 EPG 运维/华为对接人确认：

1. `openApp.jsp` 应放在哪个 **完整路径**（例如是否应为 `specialarea/yishiteng/openApp.jsp` 而非 `CNEPG/en/openApp.jsp`）。
2. 该路径下所有 `include` / `script src` 引用的文件是否已同步部署。
3. EPG 服务器 `catalina.out` / `localhost.log` 中 500 的 **Java 堆栈**（这一步能一锤定音）。

#### A-2. 查看 EPG 服务端日志（请运维执行）

在 EPG 服务器上查看访问 `openApp.jsp` 时刻的异常，常见关键字：

- `FileNotFoundException` → include 路径错误
- `ClassNotFoundException: com.huawei.iptvmw.epg.bean.info.UserProfile` → 未在 EPG 容器部署
- `JSP compile error` → JSP 语法或标签库问题

#### A-3. 用最小 JSP 做连通性测试

在 **同一部署路径** 临时放一个极简页面（需 EPG 运维配合部署）：

```jsp
<%@ page language="java" pageEncoding="UTF-8"%>
<html><body>
<script>
  // 仅测试：页面能否正常加载、STBAppManager 是否存在
  document.write("openApp test ok, STBAppManager=" + (typeof STBAppManager));
</script>
</body></html>
```

- 若仍 500 → 部署路径/容器权限问题
- 若 200 且能显示 `STBAppManager=object` → 再逐步加回 `UserProfile`、`passwordEncryption` 等依赖

#### A-4. 修正 EPG 入口链接

检查 EPG 栏目配置里生成 openApp 链接的模板，去掉重复 `userId`：

```text
错误：openApp.jsp?userId=xxx&&userId=xxx
建议：openApp.jsp?userId=xxx&returnurl=...
```

---

### 阶段 B：JSP 正常后再修拉起逻辑

#### B-1. 统一包名（三处必须一致）

| 位置 | 应填写的值 |
|------|-----------|
| APK `applicationId`（build.gradle） | 选定一个包名 |
| `openApp.jsp` 中 `appPkg` / `appName` / `className` | 与 APK 一致 |
| `judgeApk()` 中 `isAppInstalled(...)` | **与 APK 一致**（当前仍是旧包名，必须改） |

**方案 1：继续用 `com.example.myapplication`**

修改 `app/build.gradle`：

```gradle
applicationId "com.example.myapplication"
namespace 'com.example.myapplication'
```

并同步修改所有 Java 文件的 `package` 声明与目录结构（或使用 Android Studio Refactor → Rename Package）。

**方案 2：用本仓库现有包名（改动更小）**

`openApp.jsp` 全部改为：

```text
包名：com.iknet.bloodmeasuredemo
类名：com.iknet.bloodmeasuredemo.MainActivity
```

#### B-2. 修改 `judgeApk()`（必改）

```javascript
// 将旧包名 com.video.ytlook_SZDX 改为你实际 APK 的包名
function judgeApk(){
    var value = STBAppManager.isAppInstalled("com.example.myapplication");
    // 若采用本仓库包名，则写：com.iknet.bloodmeasuredemo
    return value;
}
```

#### B-3. 应用商城后台更新 appId

若走「未预装 → 应用商城下载」分支：

1. 在 `com.amt.appstore.gddx` 运营后台为新包名创建/更新应用条目。
2. 将 `appId=479` 换成 **新应用对应的 appId**（向广电/应用商城运营确认）。
3. 确认 `jumpId=8` 对该 SP 仍然有效。

#### B-4. 已预装分支的 Intent 参数

已预装时直接启动：

```javascript
STBAppManager.startAppByIntent(
  "{\"intentType\":0," +
  "\"appName\":\"com.example.myapplication\"," +
  "\"className\":\"com.example.myapplication.MainActivity\"," +
  "\"extra\":[...]}"
);
```

确认：

- APK 已安装到机顶盒（`adb shell pm list packages | grep myapplication`）
- `MainActivity` 在 Manifest 中有 `MAIN` + `LAUNCHER` intent-filter（本仓库已具备）
- 机顶盒未禁止第三方 APK 启动

| 步骤 | 操作 | 期望结果 |
|------|------|----------|

## 6. 建议修改的 openApp.jsp 片段（JSP 500 修好后使用）

以下为 **逻辑修正示例**（部署路径与 include 仍需按 EPG 平台要求调整）：

```javascript
// 【修改1】判断目标 APK 是否已安装——包名必须与 APK 一致
function judgeApk(){
    // 改成你真实安装到机顶盒上的包名
    return STBAppManager.isAppInstalled("com.example.myapplication");
}

// 【修改2】应用商城跳转参数——appPkg 与 appId 必须与商城后台一致
value: "jumpId=8&appPkg=com.example.myapplication&appId=你的新appId&account=null",

// 【修改3】已预装直启——包名、类名与 APK Manifest 一致
STBAppManager.startAppByIntent(
  "{\"intentType\":0," +
  "\"appName\":\"com.example.myapplication\"," +
  "\"className\":\"com.example.myapplication.MainActivity\"," +
  "\"extra\":[{\"name\":\"userid\",\"value\":\"" + userId + "\"}," +
  "{\"name\":\"jsessionID\",\"value\":\"" + jsessionID + "\"}," +
  "{\"name\":\"epgDomain\",\"value\":\"" + EPGDomain + "\"}," +
  "{\"name\":\"spID\",\"value\":\"szmg\"}]}"
);
```
