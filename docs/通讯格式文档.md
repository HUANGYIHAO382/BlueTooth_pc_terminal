## 2. 传输层约定（所有格式共用）

| 项 | 约定 |
|----|------|
| 协议 | **UDP**（无连接；每个包独立） |
| 编码 | **UTF-8** |
| 单包上限 | 建议 ≤ 4 KB（实际消息远小于此） |
| 默认端口 A | **18500**（发现 + 遥测；L0/T0 时也可双向控制） |
| 默认端口 B | **18501**（P0 血压控制闭环） |
| PC 发送广播 | 目的地址 **`255.255.255.255:18500`**（仅发送用，不能用来 listen） |
| PC 监听 | **`0.0.0.0:18500`** 或 **`0.0.0.0:18501`** |
| 单播加固 | PC→TV 建议同时 **广播 + 单播 TV_IP:18500** |

**真机联调地址示例：**

| 角色 | 示例 IP | 说明 |
|------|---------|------|
| PC | `192.168.1.100` | `ipconfig` 查看，写入 `script_ip` |
| TV / 真机 | `192.168.1.50` | TV 设置 WLAN IP，作单播目标 |
| AS 模拟器访问宿主机 | `10.0.2.2` | TV 发指令到 PC 时用 |
| PC 发模拟器（adb forward） | `127.0.0.1` | PC 单播目标 |

## 3. 格式族与阶段对照

同一 UDP 端口上，历史上会出现 **三种包体形态**：

| 格式族 | 阶段 | 包体形态 | TV 解析方式 |
|--------|------|----------|-------------|
| **F1 · 纯文本行** | **L0（现网 PC 脚本默认）** | 一行可读中文 + 时间戳 | 整包当字符串显示 |
| **F2 · 扁平 JSON** | L0 备选 / mock / 过渡 | `{"type":"...", ...}` 单层字段 | 按 `type` 路由 |
| **F3 · 统一信封 JSON** | **P1（规划）** | `v` + `type` + `payload` 嵌套 | 先读 `v`、`type`，再读 `payload` |

**P0 产品协议**使用 **F2 扁平 JSON**（`SCRIPT_READY`、`MEASURE_PROGRESS` 等）；与现网 **F1 文本** 字段名不同，需分阶段实现。

```
阶段演进（包格式）：

  L0   F1 纯文本（18500）          ← pc_ble_client 今天
        ↓ 可选 F2 双发
  T0   F2 JSON 上 18500 + 仍可 F1
        ↓
  P0   F2 JSON：A=18500，B=18501 闭环
        ↓
  P1   F3 统一信封 + 心跳
```

## 4. F1 · 纯文本行（L0 现网，`pc_ble_client`）

### 4.1 适用条件

- PC 底部勾选「推送心率/血压到 TV」
- `tv_link.text_mode = True`（默认）
- 端口：**18500**（单播或广播）
- **每个 UDP 包 = 一整行文本**，不是 JSON

### 4.2 行格式规范

```
[HH:MM:SS] <业务文本>
```

| 部分 | 说明 | 示例 |
|------|------|------|
| 时间戳 | 24 小时制，方括号包裹 | `[16:58:20]` |
| 业务文本 | PC 已处理好的可读句，含单位 | `心率: 74 BPM` |

**注意：** 时间戳在 **行内**；没有 `mac`、没有原始 hex、没有 JSON 外壳。

### 4.3 PC → TV 文本样例（按业务）

| 场景 | 整包内容示例 |
|------|----------------|
| 心率实时 | `[15:49:06] 心率: 74 BPM` |
| 血压加压中 | `[16:58:20] 加压中: 97 mmHg` |
| 血压结果 | `[16:58:35] 血压: 131/80 mmHg，脉搏 68 BPM` |
| TV 联动就绪（可选） | `[16:55:00] 瑞光电子血压计 就绪，等待开始测量` |
| 测试连接 PING | `[16:50:00] PING 测试连接` |

### 4.4 TV → PC 文本（L0 触发测量）

| 整包内容 | 说明 |
|----------|------|
| `START` | 纯文本，无 JSON |
| 发送到 | **`PC_IP:18500`**（与 PC listen 同端口） |

PC 收到后进入测量流程（若已开 TV 联动或兼容逻辑）。

### 4.5 TV 端解析建议（L0）

1. `bytes.decode("utf-8").strip()` 得到一行字符串。
2. 若以 `[` 开头且含 `]`，可按 `]` 后空格切开时间戳与正文（可选）。
3. **直接追加到 UI 信息流**，无需 `json.loads`。
4. 若 `json.loads` 失败，**应回退为文本行处理**（兼容 L0）。

## 5. F2 · 扁平 JSON（L0 备选 / T0 / P0）

### 5.1 通用规则

- 包体：**一个 JSON 对象**，`json.dumps(..., ensure_ascii=False)`
- 路由键：顶层字段 **`type`**（字符串，大小写见各表）
- 时间戳：
  - P0 发现/遥测常用 **`timestamp`**（毫秒）
  - 现网 `tv_link` 部分消息用 **`ts`**（秒）——过渡时需兼容

**判断是否为 JSON：** 包体以 `{` 开头且 `json.loads` 成功 → F2；否则按 F1 文本处理。

### 5.2 现网 PC 脚本 JSON（`text_mode=False` 时，Legacy 工程协议）

实现参考：`pc_ble_client/tv_link.py`。与 P0 的 `type` 命名 **不完全相同**。

#### PC → TV（默认端口 18500）

| type | 方向 | 字段 | 说明 |
|------|------|------|------|
| `READY` | PC→TV | `device`, `mac`, `name`, `ts` | 联动：设备就绪，等待 TV 授权 |
| `HR` | PC→TV | `device`, `mac`, `bpm`, `text`, `ts` | 心率（`text` 为展示行，可选） |
| `PRESSURE` | PC→TV | `device`, `mac`, `mmhg`, `text`, `ts` | 加压过程 |
| `RESULT` | PC→TV | `device`, `mac`, `data`, `text`, `ts` | 血压结果 |
| `PING` | PC→TV | `ts` | 测试连接 |

**`RESULT` 的 `data` 结构：**

```json
{
  "type": "RESULT",
  "device": "bp",
  "mac": "A4:C1:38:55:39:2A",
  "data": { "sys": 131, "dia": 80, "pulse": 68 },
  "text": "[16:58:35] 血压: 131/80 mmHg，脉搏 68 BPM",
  "ts": 1781423295
}
```

**`HR` 样例：**

```json
{
  "type": "HR",
  "device": "band",
  "mac": "F8:3D:7E:09:BD:BB",
  "bpm": 74,
  "text": "[15:49:06] 心率: 74 BPM",
  "ts": 1781421780
}
```

#### TV → PC（Legacy：18500；P0 目标：18501）

| type | 字段 | 说明 |
|------|------|------|
| `START` | `device`（可选，如 `"bp"`） | 触发 PC 开始测量 |
| `PONG` | `ts` | 回应 PING |

**`START` 样例：**

```json
{ "type": "START", "device": "bp" }
```

纯文本 `START` 与上述 JSON 等价（L0）。

### 5.3 P0 / T0 产品协议 JSON（与 mock_gateway 对齐）

以下 `type` 为 TV 产品化目标格式，见 [udp_信道设计.md](./udp_信道设计.md)、[pc_gateway升级方案.md §6](./pc_gateway升级方案.md)。

#### 信道 A · 18500 · PC → TV

| type | 频率 | 主要字段 |
|------|------|----------|
| `SCRIPT_READY` | 启动 + 约 60s | `script_ip`, `listen_port`, `devices[]`, 可选 `device_type` |
| `GATEWAY_HEARTBEAT` | 约 60s（P1） | `ts`, `uptime` |
| `HEART_RATE_STREAM` | 约 1 Hz | `heart_rate`, `timestamp`, `device` |
| `HR_SIGNAL_PRESENT` | 约 10s | `last_bpm` 等 |
| `HEART_RATE_IDLE` | 事件 | `reason` |
| `DEVICE_READY` | 事件 | `device`, `device_name`, `timestamp` |
| `DEVICE_OFFLINE` | 事件 | `device`, `reason` |

**`SCRIPT_READY` 样例：**

```json
{
  "type": "SCRIPT_READY",
  "script_ip": "192.168.1.100",
  "listen_port": 18501,
  "device_type": "BP",
  "devices": ["BP", "Band"]
}
```

**`HEART_RATE_STREAM` 样例：**

```json
{
  "type": "HEART_RATE_STREAM",
  "timestamp": 1718000000000,
  "heart_rate": 74,
  "device": "Band"
}
```

**`DEVICE_READY` 样例：**

```json
{
  "type": "DEVICE_READY",
  "device": "BP",
  "device_name": "瑞光血压计",
  "timestamp": 1718000000000
}
```

**T0 说明：** `MEASURE_PROGRESS` / `MEASURE_RESULT` **可先走 18500**；P0 目标改走 **18501**。

#### 信道 B · 18501 · TV ↔ PC（血压闭环）

| type | 方向 | 主要字段 |
|------|------|----------|
| `START_MEASURE` | TV→PC | `request_id`, `target_device`, `reply_to` |
| `CANCEL_MEASURE` | TV→PC | `request_id`, `target_device` |
| `ACK` | PC→TV | `request_id`, `message` |
| `MEASURE_PROGRESS` | PC→TV | `request_id`, `device_category`, `phase`, `progress`, `pressure_mmhg` |
| `MEASURE_RESULT` | PC→TV | `request_id`, `device_category`, `payload` |
| `MEASURE_ERROR` | PC→TV | `request_id`, `error_code`, `message` |

**TV → PC `START_MEASURE` 样例：**

```json
{
  "type": "START_MEASURE",
  "request_id": "1718000000123",
  "target_device": "BP",
  "reply_to": { "ip": "192.168.1.50", "port": 18501 }
}
```

**PC → TV `ACK` 样例：**

```json
{
  "type": "ACK",
  "request_id": "1718000000123",
  "message": "BP started"
}
```

**PC → TV `MEASURE_PROGRESS` 样例：**

```json
{
  "type": "MEASURE_PROGRESS",
  "request_id": "1718000000123",
  "device_category": "BP",
  "phase": "inflating",
  "progress": 60,
  "pressure_mmhg": 97
}
```

| `phase` 值 | 含义 |
|------------|------|
| `inflating` | 加压 |
| `deflating` | 减压 |
| `analyzing` | 分析中 |

**PC → TV `MEASURE_RESULT` 样例：**

```json
{
  "type": "MEASURE_RESULT",
  "request_id": "1718000000123",
  "device_category": "BP",
  "payload": {
    "systolic": 131,
    "diastolic": 80,
    "pulse": 68,
    "timestamp": 1718000000000
  }
}
```

**PC → TV `MEASURE_ERROR` 样例：**

```json
{
  "type": "MEASURE_ERROR",
  "request_id": "1718000000123",
  "device_category": "BP",
  "error_code": "TIMEOUT",
  "message": "测量超时"
}
```

**`request_id`：** 一次测量全链路唯一，TV 与 PC 日志用同一 ID 关联。

## 6. F3 · 统一信封 JSON（P1 规划，可选）

P1 阶段建议所有 F2 消息升级为统一外壳（见 [udp_信道设计 §5](./udp_信道设计.md)）：

```json
{
  "v": 1,
  "type": "HEART_RATE_STREAM",
  "msg_id": "uuid-or-ts",
  "ts": 1718000000000,
  "device": "Band",
  "payload": {
    "heart_rate": 72
  }
}
```

| 字段 | 说明 |
|------|------|
| `v` | 协议版本；不兼容时拒绝并提示升级 |
| `type` | 与 F2 相同语义 |
| `msg_id` | 去重、日志关联 |
| `ts` | 毫秒时间戳 |
| `device` | `Band` / `BP` / `Gateway` |
| `payload` | 业务字段（原 F2 除 type 外的内容） |

信道 B 控制类在 `payload` 内携带 `request_id`、`reply_to` 等。

## 7. Legacy 文本 ↔ P0 JSON 字段映射（PC 内部）

PC 不应让 TV 用正则解析中文。在 PC 测压循环里做转换（见 [pc_gateway升级方案 §7](./pc_gateway升级方案.md)）：

| Legacy 文本行 | 目标 JSON type | 目标字段 |
|---------------|----------------|----------|
| `[t] 心率: 74 BPM` | `HEART_RATE_STREAM` | `heart_rate: 74` |
| `[t] 加压中: 97 mmHg` | `MEASURE_PROGRESS` | `pressure_mmhg: 97`, `phase`, `progress` |
| `[t] 血压: 131/80 mmHg，脉搏 68 BPM` | `MEASURE_RESULT` | `systolic`, `diastolic`, `pulse` |

## 8. TV 端解析流程（推荐）

```
收到 UDP 包 (bytes)
    │
    ├─ 尝试 json.loads
    │     ├─ 成功 → 读 type 字段
    │     │         ├─ SCRIPT_READY / HEART_RATE_STREAM / …  → P0/T0 路由
    │     │         ├─ READY / HR / RESULT / …               → Legacy JSON 路由
    │     │         └─ START / START_MEASURE / …               → 控制指令
    │     └─ 失败 → 当作 F1 纯文本行，直接显示
    │
    └─ 根据来源端口（18500 / 18501）辅助判断阶段
```

**同一端口 T0 可能双发：** 先收到文本行、再收到 JSON，或只发其一——以 JSON 驱动图表，文本仅调试。

## 10. 常见问题

| 现象 | 原因 | 处理 |
|------|------|------|
| TV 收到一大串 `{...}` | PC 在发 F2 JSON，TV 当字符串显示 | L0 应让 PC 开 `text_mode`；或 TV 解析 JSON |
| TV 收不到包 | 网段/防火墙/未单播 | 同 WiFi；PC 单播 `TV_IP:18500` |
| `START_MEASURE` 无反应 | PC 仅 L0，未 listen 18501 | 先用 L0 `START`→18500，或 PC 升级 P0 |
| 同一秒两条（文本+JSON） | T0 双发 | 正常；UI 用 JSON，文本进工程师日志 |
| `READY` vs `SCRIPT_READY` | 工程协议 vs 产品协议 | 见 §5.2 与 §5.3，勿混 type |
